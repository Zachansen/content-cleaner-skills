# Pattern-Based Generator

Generate content that is deeply informed by a pattern library, producing output that reflects the actual methodology, language, and style of the source material rather than generic AI responses.

## Why This Exists

Standard AI content generation pulls from the model's general training data. If you ask Claude to "write a hypnosis script," you get a generic hypnosis script. If you ask it to "write a sales email," you get generic sales copy.

This skill forces generation to be rooted in a specific pattern library. Every framework referenced, every phrase used, every story told comes from the actual source material. The result is content that sounds like IT CAME FROM the person whose knowledge was extracted, not from a generic AI.

## Prerequisites

This skill requires a `pattern-library.md` file in the current project directory (or a path specified by the user). This file is generated by the pattern-library-builder skill.

If no pattern library exists, tell the user they need to:
1. Process their transcripts with the `knowledge-extractor` skill
2. Consolidate the extractions with the `pattern-library-builder` skill
3. Then come back to generate content

## How To Use

### Step 1: Load the Pattern Library

When the user requests content generation:

1. Look for `pattern-library.md` in the current directory
2. If not found, ask the user for the path
3. Read the entire pattern library into context
4. Confirm you've loaded it and briefly state what you found (e.g., "Loaded pattern library for [name]. Found 12 core frameworks, 45 language patterns, and 8 signature stories.")

### Step 2: Clarify the Content Request

Before generating, confirm:

1. **What type of content?** (script, email, sales page, training module, social post, workshop outline, etc.)
2. **What's the topic/focus?** (specific subject within the methodology)
3. **What domain(s)?** (check the Domain Index in the pattern library and suggest relevant domains. e.g., "I see domains for anxiety, confidence, and sleep — which applies here?")
4. **Who's the audience?** (beginners, advanced, specific niche)
5. **What length?** (approximate)
6. **Any specific frameworks or stories they want included?** (reference by name from the pattern library)

Ask these one at a time if the user hasn't provided them. Don't dump all questions at once.

Once a domain is selected, filter the pattern library to prioritize:
- Items tagged with that domain
- Items tagged `[domains: universal]`
- Deprioritize (but don't exclude) items from unrelated domains

### Step 3: Generate Using the Constraint System

This is the critical part. When generating content, follow these hard constraints:

#### MUST Use (Non-Negotiable)
- **Frameworks:** Only reference frameworks from the pattern library. Never invent new frameworks or pull from general knowledge.
- **Language patterns:** Use the signature phrases from the library. Don't paraphrase them into generic language.
- **Definitions:** When a term appears in the Definitions Dictionary, use THEIR definition.
- **Belief system:** Generated content must align with the principles in the pattern library. Never contradict them.

#### SHOULD Use (Strongly Preferred)
- **Stories:** Pull from the Story Bank when examples are needed. Reference them naturally, not like you're reading from a database.
- **Objection handling:** When addressing resistance or objections, use the reframes from the library.
- **[CORE] elements first:** Prioritize items tagged [CORE] over [RECURRING] over [SINGLE].

#### MUST NOT Do
- **Don't invent frameworks** that aren't in the pattern library
- **Don't use generic AI language** when a specific signature phrase exists for that concept
- **Don't make up stories or case studies** that aren't in the Story Bank
- **Don't contradict the belief system** even if common industry advice differs
- **Don't water down strong positions** the source material takes a clear stance on

### Step 4: Self-Check Before Delivering

Before presenting the generated content, run this internal checklist:

1. Does every framework referenced exist in the pattern library? If not, remove or replace.
2. Are signature phrases used naturally (not forced in awkwardly)?
3. Does the tone match the language patterns in the library?
4. Would the original person recognize this as something they'd say or create?
5. Are there any generic AI-sounding phrases that could be replaced with language from the library?

If any check fails, revise before delivering.

## Content Type Templates

Below are generation guidelines for common content types. Use these as starting structures, but always defer to the pattern library for actual content.

### Scripts (Hypnosis, Coaching, Training)
- Open with a technique from the Frameworks section
- Use transition patterns from the Language Reference
- Incorporate specific language patterns throughout (not just in key moments)
- Close with methodology-consistent wrap-up
- Include stage directions or facilitator notes where appropriate

### Sales/Marketing Copy
- Lead with a belief from the Belief System
- Use the Objection & Reframe Library for handling resistance
- Reference stories naturally as proof points
- Use signature phrases in headlines and key moments
- Match the tactical advice for whatever is being sold

### Training/Workshop Content
- Structure around core frameworks
- Use stories as teaching moments
- Include the definitions dictionary terms with their specific definitions
- Build exercises around the tactical playbook
- Address likely objections proactively using the reframe library

### Email Sequences
- Each email should lean on ONE core framework or principle
- Use language patterns in subject lines and hooks
- Stories work as email opens or proof sections
- Tactical advice makes good "here's what to do" sections
- Signature phrases as P.S. lines or callbacks

## Output Handling

- Always output generated content as a markdown file
- Name it descriptively: `[content-type]-[topic]-generated.md`
- Include a metadata header:

```markdown
---
Generated from: [pattern-library filename]
Content type: [what was generated]
Date: [YYYY-MM-DD]
Core frameworks used: [list]
Language patterns applied: [count]
Stories referenced: [list]
---
```

- After delivering, ask: "Want me to adjust anything? I can lean harder on specific frameworks, swap in different stories, or adjust the tone."

## Important Guidelines

- **Sound human, not templated.** The goal is content that sounds like the original person created it. Don't make it sound like a robot filling in a template.
- **Weave, don't list.** Frameworks and language patterns should be woven into the content naturally. Don't create a section that says "According to the [Framework Name]..."
- **Match energy.** If the source material is casual and direct, the generated content should be too. If it's more formal and structured, match that.
- **When in doubt, quote.** If you're unsure how the source would phrase something, default to using their actual words from the quotes in the pattern library.
- **Flag gaps.** If the user requests content on a topic that the pattern library doesn't cover well, say so. Don't fill gaps with generic content. Instead, ask if they have additional transcripts to process on that topic.
